# 多Agent合作的动态知识库系统设计

## 一、详细设计步骤

### 1. 需求分析与目标定义
- **业务场景梳理**：企业内部存在三类异构知识：制度文件（Wiki）、个人会话问答（非结构化QA历史）、员工档案（考勤、绩效、打卡等结构化数据）。
- **核心目标**：构建一个多Agent协同的知识库平台，实现异构知识的统一存储、智能检索、自动分类、内容审核优化及可视化呈现。
- **关键能力清单**：
  - 规定文件以Wiki形式组织与浏览；
  - 个人会话问答存入向量数据库，支持语义检索；
  - 员工档案类数据存入轻量关系数据库，支持精确查询与统计；
  - 跨库联合查询（如根据Wiki考勤制度和实际打卡记录计算扣款）；
  - 提交资料时自动判断存储环境、审核并优化内容；
  - 根据提示词生成结构化资料的思维导图。

### 2. 系统总体架构设计
采用 **分层协同架构**，包含四个核心层：
- **接入层**：Web提交维护页面、问答接口、提示词输入接口。
- **调度层**：中央调度Agent（Orchestrator），负责意图识别、任务分解、Agent路由与结果聚合。
- **Agent协作层**：多个专业化Agent，分别负责Wiki管理、向量检索、轻量数据库查询、审核优化、思维导图生成等。
- **存储层**：
  - Wiki引擎（如Wiki.js / DokuWiki）或自建Markdown版本库；
  - 向量数据库（Milvus / Chroma / Pinecone）；
  - 轻量数据库（SQLite / PostgreSQL / MySQL）。
- **模型层**：大语言模型提供自然语言理解、内容审核、优化改写、思维导图结构生成等能力。

### 3. 多Agent角色定义与协作机制
| Agent名称 | 职责 | 关键能力 |
|-----------|------|----------|
| 路由Agent | 接收用户查询，判断是单一库查询还是跨库联合查询，指派执行Agent | 意图分类、参数提取、任务编排 |
| Wiki Agent | 管理Wiki文档的CRUD、全文检索、版本控制 | 对接Wiki引擎API，输出Markdown/HTML |
| 向量Agent | 管理会话问答的嵌入、存储与语义搜索 | 文本嵌入、相似度匹配、上下文召回 |
| 轻库Agent | 员工档案（绩效、打卡、薪酬）的条件查询、聚合计算 | SQL生成、安全过滤、数据脱敏 |
| 审核优化Agent | 对用户提交的资料进行合规检查、内容优化、自动分类 | 敏感词过滤、格式标准化、分类推荐 |
| 思维导图Agent | 接收提示词，检索相关知识并生成思维导图结构 | 知识提取、层级组织、生成JSON/Mermaid格式 |
| 协调Agent | 处理跨库场景（如“员工本月迟到扣款”），先后调用轻库和Wiki Agent，计算规则 | 工作流引擎、规则引擎、结果整合 |

**协作流程**：路由Agent解析用户输入 → 判定任务类型 → 单一任务直接分派，复合任务交给协调Agent → 协调Agent定义子任务DAG → 各Agent执行并返回 → 聚合结果返回用户。

### 4. 知识库存储详细设计

#### 4.1 Wiki形式存储（规定性质文件）
- **存储引擎**：采用支持Markdown、版本管理、权限控制的Wiki系统（推荐Wiki.js，数据库PostgreSQL，文件存储本地或S3）。
- **数据组织**：按公司规章制度、部门手册、操作流程等创建命名空间/页面树。
- **检索方式**：Wiki内置全文搜索 + Agent通过REST API获取页面内容，转化为结构化摘要参与联合查询。

#### 4.2 向量数据库（个人会话问答信息）
- **数据模型**：每条记录包含 `{id, user_id, question, answer, timestamp, embedding}`。
- **嵌入方式**：使用BGE-large-zh或text-embedding-ada-002生成768/1536维向量。
- **索引类型**：HNSW或IVF_FLAT。
- **对话上下文处理**：将同一会话的Q&A拼接，以会话窗口为单位存入，保留对话链。
- **检索流程**：向量Agent将用户当前查询向量化，搜索Top-K相似记录，返回上下文和答案片段，结合LLM生成最终回答。

#### 4.3 轻量数据库（员工档案类数据）
- **数据库选型**：SQLite（轻量）或MySQL。
- **表结构示例**：
  - `employee` (id, name, dept, position)
  - `attendance` (id, emp_id, date, clock_in, clock_out, status)
  - `performance` (id, emp_id, year, score, level, comment)
- **查询接口**：轻库Agent将自然语言转为参数化SQL（利用LLM的SQL生成能力），严格限制权限和数据范围，禁止直接执行原生SQL，需经语法校验和权限过滤。

### 5. 智能调用与跨库查询流程设计
以“员工张三本月考勤信息和迟到扣款”为例：
1. **路由Agent**：识别包含“员工姓名”、“本月”、“考勤”、“扣款”，判断需联合Wiki和轻库。
2. **协调Agent**分解任务：
   - 子任务A：轻库Agent查询`attendance`表中张三本月的所有打卡记录，统计迟到次数、缺勤天数。
   - 子任务B：Wiki Agent检索“考勤管理制度”页面，提取迟到扣款标准（如迟到一次扣50元，三次以上扣绩效分等）。
3. **协调Agent**结合A的结果与B的规则，计算扣款金额，给出详细明细，并标注政策依据（引用Wiki条目链接）。
4. 返回最终回答，附加信息来源。

类似机制应用于其他跨域查询：绩效+制度、请假+剩余年假等。

### 6. 提交维护页面与自动分类审核优化设计
**前端页面功能**：
- 资料提交表单：标题、内容（富文本/文件上传）、拟选分类（可选）、标签。
- 实时审核反馈区：显示审核状态、优化建议、最终存储位置。

**后端处理流水线（审核优化Agent主导）**：
1. **内容接收**：用户提交原始内容。
2. **自动分类**：利用文本分类模型（微调BERT或直接使用LLM few-shot）分析内容，预测最适存储环境：
   - 政策性、流程性、长期生效的 → Wiki；
   - 个人经验问答、对话记录 → 向量库；
   - 表格化、统计类、员工个人数据 → 轻量数据库。
3. **内容审核**：检查敏感词、信息合规性、数据完整性。若涉及员工档案，验证是否有权限提交。
4. **内容优化**：
   - Wiki类文档：自动生成目录、规范Markdown格式、补充元数据（部门、版本日期）；
   - 会话数据：提取标准问答对，去重，纠正语法错误；
   - 档案数据：校验数据类型、完整性约束，对缺失字段给出提示。
5. **存储执行**：根据分类结果调用对应Agent的写入接口，返回永久链接或确认信息。
6. **页面反馈**：展示分类结果、优化前后对比、存储位置，允许用户确认或手动修改。

### 7. 思维导图生成设计
- **输入**：用户通过界面输入提示词（如“新员工入职流程”）。
- **检索增强**：思维导图Agent向Wiki Agent、向量Agent、轻库Agent发送查询，召回相关制度文档、常见问答、流程表格。
- **结构化生成**：LLM将检索到的多源信息整合，提取层次化知识点，生成思维导图结构（JSON/Mermaid语法）。
- **前端渲染**：使用Mermaid.js或Markmap将结构渲染为交互式思维导图，支持导出PNG/SVG。

### 8. 技术选型
| 层次 | 技术组件 | 选型理由 |
|------|----------|----------|
| 大模型 | GPT-4 / Claude / Qwen-72B等 | 强语义理解、SQL生成、格式转换 |
| 框架 | LangChain / AutoGen / MetaGPT | 多Agent编排、工具调用、记忆管理 |
| Wiki | Wiki.js (PostgreSQL + S3) | 开源、API丰富、Markdown友好 |
| 向量数据库 | Milvus / Chroma | 高性能相似搜索，支持元数据过滤 |
| 轻量数据库 | PostgreSQL | 同时支持关系查询和JSON扩展，可兼顾Wiki需要 |
| 嵌入模型 | BGE-M3 / text-embedding-3-large | 多语言、高召回率 |
| 前端 | Vue3 + Vite | 构建维护页面与思维导图展示 |
| 思维导图渲染 | Markmap + Mermaid | 自动布局，交互友好 |
| 权限认证 | OAuth2.0 / LDAP | 集成企业统一账号 |

### 9. 实现步骤（分阶段迭代）
**第一阶段：基础存储与单一Agent**
- 搭建Wiki系统，导入现有规章制度。
- 部署向量数据库，实现会话数据写入与语义搜索。
- 设计轻量数据库表结构，导入员工考勤、绩效样本数据。
- 开发Wiki Agent、向量Agent、轻库Agent的基本查询接口。

**第二阶段：多Agent协作与联合查询**
- 构建路由Agent与协调Agent，实现典型跨库查询（考勤扣款、绩效等）。
- 实现审核优化Agent的内容分类与格式优化能力。
- 开发提交维护页面，打通分类→审核→存储闭环。

**第三阶段：思维导图与智能优化**
- 开发思维导图生成Agent，对接检索结果和渲染引擎。
- 强化内容审核（敏感词、合规性），支持人工确认流。
- 实现Agent记忆与会话管理，支持追问和上下文保持。

**第四阶段：测试、权限与部署**
- 单元测试、集成测试，特别是SQL注入防护和向量库数据隔离。
- 基于角色的访问控制（RBAC），确保不同员工只能查询授权范围内的数据。
- 容器化部署（Docker Compose），监控与日志。

### 10. 测试与上线保障
- **安全审计**：SQL参数化、数据脱敏、Agent权限隔离。
- **数据一致性**：跨库查询时的事务控制策略（最终一致性，带来源标注）。
- **性能优化**：缓存常见制度文档的向量，预热热门SQL查询。
- **用户培训**：维护页面操作指南，提交资料的标准模板。

## 二、项目设计总结

本项目设计了一套**基于多Agent协同的动态知识库系统**，创新性地将企业内部三类核心知识——规范制度（Wiki）、非结构化会话（向量库）、结构化档案（轻量关系库）——纳入统一管理框架，并通过智能Agent之间的分工协作，实现了跨存储形态的知识检索、自动分类与内容优化。

**核心价值：**
1. **统一入口，异构存储透明化**：用户只需通过一个问答界面或提交页面，无需关心底层存储差异。路由Agent自动分析意图，精准调度对应Agent，跨库联合查询能力解决了传统知识库割裂的痛点。
2. **主动式内容治理**：引入审核优化Agent，在数据入库前自动完成分类推荐、合规检查和格式优化，保障知识库质量和规范性，大幅降低人工维护成本。
3. **实时规则计算与溯源**：结合制度文档与个体数据，系统能即时计算出迟到扣款、绩效评估等决策支持信息，并明确引用制度依据，确保结果可解释、可追溯。
4. **知识可视化生成**：基于检索增强生成（RAG）的思维导图模块，能将分散在多库中的知识碎片按主题自动组织为层次化结构，助力快速理解和培训。
5. **动态扩展性**：多Agent架构松耦合，新增存储类型或业务能力只需注册新Agent并定义协作流，不影响现有功能，可灵活应对企业知识体系的演进。

**设计亮点：**
- 采用**中央调度与专业Agent相结合**的混合协作模式，兼顾全局统筹与领域深度。
- 实现**NL2SQL与NL2Wiki**的联合查询管道，将非结构化制度规则应用于结构化数据计算。
- 构建了从**提交→分类→审核→优化→存储→检索→可视化**的完整知识生命周期管理闭环。

通过本方案，企业能够构建一个真正“活”的知识库——它不仅是资料的仓库，更是能够理解规则、融合数据、辅助决策的认知引擎。
